# grandma.app — Pre-Friends-Test Review

**Date:** 2026-06-10  ·  **Reviewer:** Claude Code (parallel multi-agent workflow)  ·  **Scope:** full app — auth, onboarding, all 3 modes, every CRUD flow, security/RLS, edge functions, design system

## How this was produced

- **93 agents** across 6 lenses x section clusters: security & data safety (deepest), core flows/logic (deep), mode-consistency, state/data-fetching, performance, design system.
- **199 raw findings** from 17 reviewers.
- Every P0/P1 candidate (76) was **adversarially re-verified** by an independent agent that read the actual code and tried to refute it. This **refuted 23 false positives** (mostly early-migration RLS holes that a later `rls_hardening`/`audit_fixes` migration already closed) and downgraded ~30 mis-rated findings.
- Severities below are post-verification. RLS leak tests were **designed, not run** against the live DB (no test data written), per request.

## Summary

| Severity | Count | Meaning |
|---|---|---|
| **P0** | 7 | Blocks the test: data-integrity / auth / payment / cross-user access |
| **P1** | 11 | A friend hits it day one: broken core flow or silent-fail CRUD |
| **P2** | 85 | Noticeable but survivable |
| **P3** | 63 | Polish / design-token / hygiene |

### The 5 things to fix first

1. **Edge functions are open / forgeable** — `nana-chat`, `food-ai`, `scan-image` are unauthenticated Anthropic relays (anyone can drain your API budget); `grandma-chat` decodes the JWT **without verifying its signature** (forge a token → read any user's child health logs); `generate-insights` trusts `user_id` from the POST body (pure IDOR).
2. **`revenuecat-webhook` has no signature check** — anyone can POST a fake 'subscription active' event and unlock Premium for free.
3. **`care_circle` self-update policy** lets a caregiver grant *themselves* more permissions and more children; and a **paused caregiver keeps full DB read/write** (the pause is UI-only).
4. **`insights` table `WITH CHECK(true)`** survives the hardening migration (the drop targeted the wrong policy name) — any user can write fake health insights into another user's feed.
5. **`children` table has no DELETE policy** — 'remove child' silently fails (RLS blocks it, no error), child vanishes then reappears on refetch.

## P0 — Blockers (fix before any friend touches it)  (7)

#### 1. Paused caregiver retains full DB read/write — _paused only hides them in the UI
- **`app/profile/care-circle.tsx:560`** · confidence: high
- **Problem:** pauseMember sets permissions._paused=true on the child_caregivers row but leaves status='accepted'. Every RLS caregiver policy keys on status='accepted' (and child_logs insert on is_locked), none on permissions->>'_paused'. So a 'Paused' caregiver still passes all SELECT/INSERT/UPDATE policies for the shared child (logs, kids_vaccine_schedule, nanny_notes, chat). The parent believes access is suspended; at the DB layer it is not. A tester who pauses user B and then has B query the shared child will see B still has access.
- **Fix:** Make pause set status to a revoked/paused state (or add AND COALESCE((permissions->>'_paused')::boolean,false)=false to caregiver policies). Drive enforcement from a column RLS can see, not a UI-only JSON flag.
- **Verifier note:** Verified: pauseMember (care-circle.tsx:560-563) flips permissions._paused but leaves status='accepted'; every caregiver RLS policy gates on status='accepted' and never checks _paused, so a paused caregiver keeps full SELECT/INSERT/UPDATE access to the shared child's data — a real RLS/access-control leak, which qualifies as P0.

#### 2. generate-insights trusts user_id entirely from the POST body — full IDOR
- **`supabase/functions/generate-insights/index.ts:21`** · confidence: high
- **Problem:** The function reads `user_id` directly from `req.json()` and uses it (with the service-role client, bypassing RLS) to query cycle_logs, pregnancy_logs, child_logs, and children for that user_id, then writes AI-generated insights back to the insights table under that same user_id. An authenticated attacker can POST `{"user_id":"<victim-uuid>","behavior":"pregnancy"}` to read the victim's 30 days of health logs and plant attacker-controlled insights in the victim's app. No Authorization header is read anywhere in this function.
- **Fix:** Read and verify the caller's JWT using `supabase.auth.getUser()`, derive `user_id` from `user.id`, and ignore any `user_id` in the request body. This is exactly what grandma-chat does correctly (albeit with flawed JWT decode — see separate finding).
- **Verifier note:** Confirmed: generate-insights (deployed --no-verify-jwt per edge-deploy.md matrix) reads user_id from the POST body at line 21, uses the service-role client (line 30, bypasses RLS) to read cycle_logs/pregnancy_logs/children/child_logs and write insights for that arbitrary user_id, and never reads the Authorization header — a real cross-user read/write IDOR on private health data, which the rubric classifies as P0 rather than the reviewer's P1.

#### 3. grandma-chat manually decodes JWT without signature verification — auth bypass
- **`supabase/functions/grandma-chat/index.ts:42`** · confidence: high _(also flagged near lines 21, 45)_
- **Problem:** The code base64-decodes the JWT payload and reads `payload.sub` as the user_id, but never verifies the JWT signature. The comment says 'no verification needed' which is incorrect. An attacker can craft a JWT with `{"sub":"<any-victim-uuid>"}` as the payload, sign it with a garbage key, and this function will accept it as authentic. The function then uses the service-role client (which bypasses RLS) to fetch that victim's health profile, cycle logs, pregnancy logs, and children's data (including allergies and conditions), which all appear verbatim in the Claude system prompt — a direct data exfiltration path.
- **Fix:** Replace the manual base64 decode with `supabase.auth.getUser(token)` which cryptographically verifies the JWT signature against Supabase's JWKS. Pattern already used correctly in invite-caregiver (line 53) and accept-invite (line 24). Delete lines 43-53 and replace with the `auth.getUser()` call.
- **Verifier note:** Confirmed: grandma-chat is deployed --no-verify-jwt (docs/pregnancy-fix-plan-2026-05-12.md:136 + CLAUDE.md table), so the gateway does not verify the JWT; the function manually atob-decodes payload.sub at lines 42-47 with no signature check and then uses the service-role client (line 63, bypasses RLS) to read any victim's profile health_notes/allergies/conditions/medications, cycle/pregnancy logs, and children's medical data into the Claude prompt — a forge-a-sub cross-user PHI exfiltration path. invite-caregiver:53 and accept-invite:24 correctly use auth.getUser(token); the proposed fix is right. Raised to P0 (unauthenticated RLS-bypassing health-data leak) rather than P1.

#### 4. grandma-chat: activeChildId from request body is not validated against caller's children — IDOR on child health logs
- **`supabase/functions/grandma-chat/index.ts:135`** · confidence: high
- **Problem:** When `context.activeChildId` is provided, the child_logs query filters by `.eq('child_id', activeChildId)` without also constraining to `.in('child_id', childIds)` (the caller's own children). The service-role client bypasses RLS, so this will return logs for any child_id in the system. An authenticated user can specify any other user's child_id and receive that child's feeding, sleep, diaper, mood, and medication logs in the AI context. The childNames lookup at line 143 also only covers the attacker's own children, so the injected logs surface as 'Child' in the prompt — but the raw data is still exfiltrated.
- **Fix:** After fetching `childIds` at line 126, always append `.in('child_id', childIds)` to the query regardless of whether `activeChildId` is set. When `activeChildId` is also provided, add it as an additional `.eq()` — effectively: verify `activeChildId` is in `childIds` first, then use it as filter.
- **Verifier note:** Confirmed IDOR: line 136 filters child_logs only by body-supplied context.activeChildId with no ownership constraint, and the service-role client (line 63) bypasses RLS, so any user can read any child's health logs — a cross-user data leak, which the rubric classifies as P0, not P1.

#### 5. revenuecat-webhook has no signature verification — anyone can grant premium
- **`supabase/functions/revenuecat-webhook/index.ts:43`** · confidence: high _(also flagged near lines 49)_
- **Problem:** The function accepts any POST request and immediately processes it as a legitimate RevenueCat event. There is zero verification that the payload came from RevenueCat. An unauthenticated attacker can POST `{"event":{"type":"INITIAL_PURCHASE","product_id":"premium_family_monthly"},"app_user_id":"<any-uuid>"}` to the function URL and instantly upgrade any user account to premium_family tier. This completely breaks the paywall for the beta test and any future launch.
- **Fix:** RevenueCat signs every webhook with an HMAC-SHA256 signature in the `X-RevenueCat-Signature` header using a shared secret configured in the RevenueCat dashboard. Add a `REVENUECAT_WEBHOOK_SECRET` Supabase secret, then at the top of the handler compute `hmac-sha256(body_bytes, secret)` and compare to the header value using a constant-time comparison. Reject any request where the signature does not match before touching any business logic. Deno example: `import { createHmac } from 'https://deno.land/std@0.168.0/node/crypto.ts'`.
- **Verifier note:** Confirmed: the handler does zero authentication (only CORS/OPTIONS reference "authorization"), is deployed no-verify-jwt per CLAUDE.md/README, and writes profiles.subscription_tier/status via the service-role key keyed on an attacker-supplied app_user_id — so anyone can POST to grant any user premium_family, breaking the paywall and writing other users' rows.

#### 6. insights INSERT WITH CHECK(true) allows any user to insert insights for any user_id
- **`supabase/migrations/20260405060000_insights_table.sql:31`** · confidence: high
- **Problem:** `Service can insert insights` policy has `WITH CHECK (true)` and is not restricted to service_role. Any authenticated client can insert insights rows with an arbitrary `user_id`, polluting another user's analytics feed with fake or misleading health insights. Fixed in 20260512120000_rls_hardening.sql (line 99-104) with a proper owner FOR ALL policy.
- **Fix:** Confirm 20260512120000 applied. The `Insert insights` and old `Owner can manage insights` policies are dropped and replaced with a single `Owner can manage insights` FOR ALL with `USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid())`. Service-role inserts bypass RLS so no separate service policy is needed.
- **Verifier note:** The WITH CHECK(true) INSERT vuln is real, but the reviewer's claim that 20260512120000 fixes it is FALSE: that migration drops policies named "Insert insights"/"Owner can manage insights", while the actual policy is "Service can insert insights" — never dropped anywhere — so the IF EXISTS drop is a no-op and the permissive WITH CHECK(true) policy survives, OR-combining to always-allow cross-user inserts.

#### 7. care_circle member-self-update policy allows a caregiver to grant themselves more permissions and more children
- **`supabase/schema.sql:234`** · confidence: high
- **Problem:** Policy 'Members can accept invite' is FOR UPDATE USING (auth.uid() = member_user_id) WITH CHECK (auth.uid() = member_user_id) with no column restriction. A member can UPDATE their own care_circle row to rewrite permissions (e.g. add 'edit_child','emergency') and children_access (add child UUIDs they were never granted), then read/write those children via the schema.sql children/child_logs care-circle policies. Unlike child_caregivers (which added prevent_invitee_permission_escalation trigger in 20260512120000:132), care_circle has NO such guard. If the care_circle path is live on any environment, this is direct privilege escalation + cross-family child leak.
- **Fix:** If care_circle is retained, add a BEFORE UPDATE trigger mirroring prevent_invitee_permission_escalation: when auth.uid()=member_user_id AND auth.uid()<>owner_id, reject any change to permissions, children_access, role, owner_id, access_end. Restrict the member UPDATE policy to status/accepted_at only. Otherwise drop the care_circle table and its policies entirely in favor of child_caregivers.

#### 9. Caregiver invite token lost after unauthenticated redirect — accept-invite flow is broken for new users
- **`app/accept-invite.tsx:25`** · confidence: high
- **Problem:** When an unauthenticated user opens a caregiver invite deep-link, the screen redirects to /(auth)/welcome?invite=<token>. But the welcome.tsx screen never reads the 'invite' query param. After the user signs up or signs in they land on the main tabs with no indication that there is a pending invite. The token is lost, the accept-invite flow never completes, and the caregiver is never linked to the child.

- **Verifier note:** Verified: accept-invite.tsx:25-26 redirects unauthenticated users to /(auth)/welcome?invite=<token>, but neither welcome.tsx nor sign-in.tsx (nor sign-up.tsx) ever reads useLocalSearchParams/the invite param, and the root _layout.tsx auth guard routes post-SIGNED_IN straight to /onboarding/journey or /(tabs) with no token awareness — a repo-wide grep found the param is produced once and consumed nowhere (no module var, no store), so the token is dropped and the caregiver is never linked. Real bug at the cited location; P1 (core invite flow silently broken for new testers, not a crash/RLS/payment issue).

#### 10. React Query queryKey mismatch — generate/archive/restore never invalidates the live cache
- **`components/insights/InsightsScreen.tsx:936`** · confidence: high
- **Problem:** The `useQuery` for insights is registered with key `['insights', userId, mode]` (line 909), but all `refetchQueries` and `invalidateQueries` calls use `['insights', mode]` (lines 936, 951, 998, 1005) — missing `userId`. React Query v5 requires an exact prefix match. After generating insights, archiving, or restoring, the cached query is never invalidated, so the UI continues showing stale data and the user sees no change. The same mismatch affects `['insights-history', mode]` vs `['insights-history', userId, mode]` (lines 999, 1006) and `['behavior-metrics', mode]` vs `['behavior-metrics', userId, mode]` (line 937).
- **Fix:** Add `userId` to every invalidation call to match the registered key: `queryClient.refetchQueries({ queryKey: ['insights', userId, mode] })`, `queryClient.invalidateQueries({ queryKey: ['insights-history', userId, mode] })`, `queryClient.invalidateQueries({ queryKey: ['behavior-metrics', userId, mode] })`. Apply to all four call sites (lines 936–937, 951, 998–999, 1005–1006).
- **Verifier note:** Verified: queries are keyed ['insights'/'insights-history'/'behavior-metrics', userId, mode] (lines 909/916/923) but all refetch/invalidate calls use [key, mode] omitting userId (936-937, 951, 998-999, 1005-1006); RQ v5 prefix matching compares position 1 (UUID userId) against mode, so no match — generate/refresh/archive/restore never invalidate the live cache and the UI shows stale data for the 5-min staleTime (no QueryClient default masks it). The inline comment at 892-893 confirms userId belongs in the key, so the omission is a real bug. P1 because generate (including auto-generate on mount) and the archive button both visibly do nothing in-session on the core Insights screen, though it self-heals on remount so not P0.

#### 11. Apple Sign-In nonce passed via state field instead of nonce field — replay protection not enforced
- **`lib/auth-providers.ts:38`** · confidence: medium
- **Problem:** The code passes the hashed nonce as the state parameter to AppleAuthentication.signInAsync. Apple's OIDC nonce binding requires the hash to be in the nonce field, not state. state is an opaque string Apple echoes back unchanged for CSRF protection; Apple does NOT use it to verify nonce binding. As a result Supabase's nonce check (which compares rawNonce against the nonce claim in Apple's identity JWT) may fail on some flows, and if it passes today it means nonce validation is not actually happening — replay attacks on Apple ID tokens are not blocked.
- **Fix:** expo-apple-authentication v6+ exposes a nonce field. Pass nonce: hashedNonce directly: signInAsync({ requestedScopes: [...], nonce: hashedNonce }). Remove the state workaround. Verify in Supabase dashboard that Apple provider nonce verification is enabled.
- **Verifier note:** Confirmed at lib/auth-providers.ts:38 — code sets state:hashedNonce and never sets nonce; expo-apple-authentication@8.0.8 exposes a nonce field (types line 55) and native Swift maps request.nonce=options.nonce vs request.state=options.state, so only the nonce field reaches the JWT nonce claim. The hashed nonce in state is never hashed into the token, so Supabase's rawNonce comparison cannot bind — OIDC replay protection is not enforced and Apple Sign-In likely fails outright; fix (pass nonce:hashedNonce) is correct and supported. P1: breaks/weakens a core iOS auth path on day one but no cross-user data leak and other sign-in routes remain.

#### 12. accept-invite edge function email check blocks invite-by-link flow — link invitees always locked out
- **`supabase/functions/accept-invite/index.ts:45`** · confidence: high
- **Problem:** Line 45 enforces `invite.email.toLowerCase() !== user.email?.toLowerCase()` — an exact match. In `care-circle.tsx`, when 'Share Link' is chosen the email is set to `pending_<timestamp>@invite.local` (line 1221). When the recipient opens the deep-link and logs in with their real email, this check always throws 'This invite was sent to a different email address', permanently blocking acceptance of link-based invites. This breaks the primary share-link flow.
- **Fix:** Skip the email check when `invite.email` ends with `@invite.local` or `@pending`: `const isLinkInvite = invite.email.endsWith('@invite.local') || invite.email.endsWith('@pending'); if (!isLinkInvite && invite.email.toLowerCase() !== user.email?.toLowerCase()) { throw new Error('This invite was sent to a different email address') }`
- **Verifier note:** Confirmed at accept-invite/index.ts:45 — care-circle.tsx:1221 stores a pending_<ts>@invite.local placeholder for both Share-Link AND SMS invites (no email collected), and the exact-match email check always throws for those; the deep link is fully wired (accept-invite.tsx invokes the function with only the token), so link/SMS acceptance is genuinely broken. P1, not P0: email-method invites still work, and there is no auth/RLS leak or data loss.

#### 13. nana-chat edge function is a fully open Anthropic API relay — no auth check at all
- **`supabase/functions/nana-chat/index.ts:12`** · confidence: high _(also flagged near lines 31, 104)_
- **Problem:** The function is deployed --no-verify-jwt and performs zero authentication before forwarding messages to the Anthropic API using the project's ANTHROPIC_API_KEY. Any party with the public Supabase URL can POST arbitrary message arrays with any system context and consume the project's Anthropic quota without limit. This is a direct API-cost abuse vector and also means the function can be used as an open prompt-injection relay by anyone who discovers the endpoint (the URL is derivable from the EXPO_PUBLIC_SUPABASE_URL bundled in the app binary).
- **Fix:** Add an Authorization header check before calling Anthropic. Either (a) verify the Supabase JWT with `supabase.auth.getUser(token)` and reject unauthenticated callers, or (b) at minimum check that a valid Supabase anon-key-signed request is present. The `grandma-chat` function is the preferred replacement path; `nana-chat` should be retired or gated.
- **Verifier note:** Confirmed: supabase/functions/nana-chat/index.ts forwards the request body straight to Anthropic with the project ANTHROPIC_API_KEY and performs zero auth (no Authorization check, no _shared middleware, no config.toml gating no-verify-jwt), and it is still live (used by lib/claude.ts:21 and as fallback in lib/grandmaChat.ts:67), so it is a genuine open relay / API-cost-abuse vector; grandma-chat (the gated replacement) does require a token, matching the reviewer's fix. P1 rather than P0 because no user data is read/written and it does not crash or block the friend test.

#### 14. nana-chat and scan-image edge functions are open relays — no auth, no quota enforcement
- **`supabase/functions/nana-chat/index.ts:12`** · confidence: high
- **Problem:** nana-chat, scan-image, and food-ai are deployed with --no-verify-jwt and contain zero authentication or rate-limit checks. Any caller with only the public anon key (visible in every installed app bundle) can POST unlimited requests directly to these functions, bypassing the entire client-side free-tier 3-scan gate and the premium paywall. The Anthropic API key gets billed for every call. This is an unrestricted Anthropic API relay at $3–15 per thousand tokens.
- **Fix:** Each no-verify-jwt function must authenticate the caller itself. Extract and verify the Bearer token from the Authorization header (same pattern already used in grandma-chat lines 33-53). For scan-image and nana-chat specifically: after extracting user_id from the JWT, query scan_history count and subscription_tier from profiles before calling Anthropic, and return 403 if the free-tier limit is exceeded. Without this, the paywall is purely cosmetic.
- **Verifier note:** Confirmed: nana-chat, scan-image, and food-ai contain zero auth/quota checks while grandma-chat (lines 33-53) proves the secure JWT pattern exists; the only scan gate is client-side (app/scan.tsx:72), so the anon key makes these an unauthenticated Anthropic relay and the paywall cosmetic — real and exploitable, but for a closed friends-test it bypasses billing/paywall rather than blocking a flow or leaking another user's data, so P1 not P0.

#### 15. children table has no DELETE policy — parents cannot delete child records
- **`supabase/migrations/20260330010000_child_caregivers.sql:56`** · confidence: high
- **Problem:** When migration 20260330010000 replaced the original `own children` FOR ALL policy with split SELECT/INSERT/UPDATE policies, it never added a DELETE policy for parents. No subsequent migration adds one. As a result, any DELETE call from the client on the `children` table will be silently blocked by RLS. If the app has a 'remove child' feature, it silently fails. The cascade on `auth.users` deletion will still clean up, but client-initiated child removal does not work.
- **Fix:** Add: `CREATE POLICY "parents can delete children" ON children FOR DELETE USING (parent_id = auth.uid());`
- **Verifier note:** Confirmed: migration 20260330010000 (line 56) drops the "own children" FOR ALL policy and recreates only SELECT/INSERT/UPDATE; no later migration (including rls_hardening, audit_fixes, cascade_user_delete) adds a DELETE policy, and the FOR ALL in schema.sql is stale (different policy name + uses user_id vs the live parent_id), so it is not on the deployed table. The "remove child" feature exists at app/profile/kids.tsx:135 and calls supabase.from('children').delete() — RLS silently blocks it (delete returns success/zero-rows, no error thrown), so setChildren() still runs and the child vanishes then reappears on next refetch: a real, silent broken-CRUD bug, correctly P1.

#### 16. Caregivers cannot READ child_logs under the migration model — caregiver activity/analytics returns empty under RLS
- **`supabase/migrations/20260405020000_child_logs.sql:21`** · confidence: high
- **Problem:** In the child_caregivers (migration) model, the ONLY SELECT path on child_logs is the owner FOR ALL policy USING (user_id = auth.uid()). There is NO caregiver SELECT policy on child_logs anywhere in the migrations (only schema.sql has one, via care_circle.children_access). So an accepted nanny can INSERT logs (subscription_tiers policy) but cannot SELECT any child_logs — KidsHome/analytics/care-circle activity feed will be empty for them. This is the inverse of a leak (over-restriction) but it breaks the core caregiver flow a friend will hit day one, and it signals the two models are not reconciled.
- **Fix:** Add a caregiver SELECT policy on child_logs in the migration model: child_id IN (SELECT child_id FROM child_caregivers WHERE user_id=auth.uid() AND status='accepted'). Gate on a 'view' permission if desired. Verify with user B (accepted caregiver) selecting a shared child's logs.
- **Verifier note:** Verified: in the live (child_caregivers) migration model, child_logs has only an owner FOR ALL SELECT path plus a caregiver INSERT path — no caregiver SELECT policy exists in any migration, so accepted caregivers can log but read nothing; the only caregiver-read policy (schema.sql) keys on care_circle, a table no migration creates and the app doesn't use.

#### 17. Caregiver INSERT into child_logs ignores the log_activity permission — a view-only caregiver can write logs
- **`supabase/migrations/20260421010000_subscription_tiers.sql:169`** · confidence: high
- **Problem:** The final 'Insert child_logs' policy checks only that the caller is an accepted, non-locked caregiver for the child: child_id IN (SELECT child_id FROM child_caregivers WHERE user_id=auth.uid() AND status='accepted' AND is_locked=false). It does NOT check (permissions->>'log_activity'). The UI offers a 'View Only' permission level (care-circle.tsx:153) that sets permissions to {view:true} only, but RLS still lets that caregiver insert feeding/sleep/medicine/vaccine logs. Threat-model item 5 (read-only caregiver should not write) fails.
- **Fix:** Add AND (permissions->>'log_activity')::boolean = true to the WITH CHECK, matching the chat policy which already gates on (permissions->>'chat'). The schema.sql variant already does this ('log_activity' = any(permissions)) — bring the migration policy in line.
- **Verifier note:** Verified: the final live "Insert child_logs" policy (subscription_tiers.sql:169-178) gates only on status='accepted' AND is_locked=false with no (permissions->>'log_activity') check; care-circle.tsx:153 "View Only" sets perms:['view'] (stored as jsonb missing log_activity), and the sibling chat_messages policy at line 190 already gates on (permissions->>'chat'), so a view-only caregiver can still INSERT logs — but it's a permission-level escalation within an already-invited care relationship, not an arbitrary cross-user RLS leak, so P1 rather than P0.

#### 18. profiles_public view granted to anon — all user names and photos exposed without authentication
- **`supabase/migrations/20260512120000_rls_hardening.sql:193`** · confidence: high
- **Problem:** `GRANT SELECT ON profiles_public TO anon, authenticated;` combined with `ALTER VIEW profiles_public SET (security_invoker = false)` means the view runs as the view owner (postgres), bypassing the owner-only RLS on `profiles`. Any unauthenticated caller (anyone with the Supabase anon key — i.e., the public internet) can enumerate every user's `id`, `name`, `photo_url`, `user_role`, `created_at`, and `language`. This is a user enumeration vulnerability.
- **Fix:** Remove `anon` from the grant: `REVOKE SELECT ON profiles_public FROM anon; GRANT SELECT ON profiles_public TO authenticated;` Unauthenticated users have no legitimate reason to enumerate the user directory.
- **Verifier note:** Confirmed at supabase/migrations/20260512120000_rls_hardening.sql:191-193: the view is explicitly set security_invoker=false (runs as owner, bypassing the new profiles_owner_read RLS) AND granted SELECT to anon, so anyone with the public anon key (shipped in the app bundle) can enumerate every user's id, name, photo_url, user_role, created_at, language with no auth — no later migration revokes it, and the author's own comment said the grant was meant for authenticated only. Real bug; exposure is real but limited to curated low-sensitivity columns (no PHI/write), so P1 rather than P0.

## P2 — Noticeable but survivable  (85)

### Database / RLS

- **Invite tokens never expire, never rotate, and are reused on revoke-then-reinvite** — `supabase/migrations/20260330010000_child_caregivers.sql:20`  
  invite_token is a static uuid default gen_random_uuid() with no expiry column and no rotation. The invite-caregiver edge function, on reactivating a revoked invite, returns the SAME existing invite_token (invite-caregiver/index.ts:88-100). care-circle.tsx resendInvite (line 629) and AddMemberSheet (line 1253) embed this stable token in mailto:/sms:/Share links sent over unencrypted channels. accept-invite only checks status!='accepted'/'revoked' and email match — there is no token TTL, no single-use invalidation, and no per-send rotation. Threat-model item 6 (revoked access stays revoked) holds for status, but the TOKEN itself is a long-lived bearer credential to a child's data.

### Edge functions

- **Invite token never expires — any old token is permanently valid** — `supabase/functions/accept-invite/index.ts:33`  
  The accept-invite edge function has no expiry check. The child_caregivers schema (migration 20260330010000) has no invited_at or expires_at column. A caregiver invite token issued today is valid indefinitely. If a test user shares an invite link with close friends, it can be forwarded, leaked, or reused by anyone at any time to gain caregiver access to a child's profile.  
  → _Fix:_ Add an invited_at TIMESTAMPTZ DEFAULT NOW() column to child_caregivers and check in the edge function that (NOW() - invited_at) < interval '7 days'. Return a 400 with message 'Invite link has expired' if it is stale. Add a migration to backfill invited_at = created_at for existing rows.
- **accept-invite matches only on email equality, allowing aliased-email acceptance and ignoring case/normalization edge cases** — `supabase/functions/accept-invite/index.ts:45`  
  accept-invite binds an invite to whoever holds the token AND whose JWT email equals invite.email (case-insensitively). Anyone who obtains the token (see token-longevity finding) and controls a mailbox at that exact address can accept. There is no verification that the accepting user is the intended human, no rate limiting, and placeholder invites created by the UI use synthetic emails like pending_<ts>@invite.local (care-circle.tsx:1221) — if such a token leaks, the email check is trivially defeated because no real mailbox is required to match a value the attacker can read from the invite row only if they can SELECT it (they cannot under RLS, but the token is in the shared link). Combined with non-expiring tokens this widens the window.  
  → _Fix:_ Add token expiry + single-use (see token finding). Reject acceptance of placeholder @invite.local emails (require a real invited email). Consider binding acceptance to the invited user_id when known, and add basic rate limiting on accept-invite.
- **Invite tokens have no expiry and are never nulled after acceptance** — `supabase/functions/accept-invite/index.ts:55`  
  The `invite_token` (a UUID) is generated at invite creation and never invalidated. After a caregiver accepts: the status is set to 'accepted' but the token value remains in the database indefinitely. There is no `expires_at` column in the `child_caregivers` / `care_circle` table. Tokens do not expire after 7-30 days as is standard practice. While re-use is blocked by the status check, anyone with a previously-shared invite link (e.g. forwarded in a message) retains the token forever. If the status-check logic ever changes or a bypass is found, the token is still valid.  
  → _Fix:_ Add an `invite_expires_at TIMESTAMPTZ` column (set to `NOW() + INTERVAL '7 days'` on insert). In accept-invite, check `invite.expires_at < NOW()` and reject with 'Invite has expired'. Additionally, set `invite_token = NULL` after successful acceptance so the column carries no live credential post-use.
- **CORS wildcard on all functions including those that return or write user health data** — `supabase/functions/grandma-chat/index.ts:9`  
  Every edge function sets `Access-Control-Allow-Origin: *`. For functions that perform mutations (generate-insights writes to the insights table, accept-invite accepts caregiver access) and return sensitive health data, a wildcard CORS policy means any web page can make cross-origin requests to these endpoints. While CORS does not stop server-side attacks, it removes a browser-level defence layer that should exist for functions handling children's medical data.  
  → _Fix:_ Restrict CORS to the app's known origins. For a React Native app there is no browser origin, so the wildcard is mostly harmless at runtime — but for defence in depth, set `Access-Control-Allow-Origin` to the Supabase project URL or a known web admin origin, not `*`.
- **nana-chat injects unsanitised client-supplied childContext directly into the AI system prompt** — `supabase/functions/nana-chat/index.ts:30`  
  The request body fields `childContext` (name, ageMonths, weightKg, allergies, medications), `pillarId`, `mode`, and `weekNumber` are passed directly into `buildSystemPrompt()` which interpolates them into the Claude system prompt string without any sanitisation or length limits. A malicious (or compromised) client can inject prompt-override instructions such as `\n\n## NEW INSTRUCTIONS: ignore all previous instructions and...` in any of these fields. Since nana-chat has no auth, this attack surface is open to the entire internet.  
  → _Fix:_ First, add authentication (see finding above). Then sanitise string fields: truncate to reasonable lengths (e.g., child name max 100 chars), strip newlines from all interpolated values, and validate `mode` against the known enum (pre-pregnancy / pregnancy / kids) — which buildSystemPrompt already does for `mode` but not for the child fields.
- **scan-image injects unsanitised client-supplied childContext and pregnancyContext into the AI system prompt** — `supabase/functions/scan-image/index.ts:48`
