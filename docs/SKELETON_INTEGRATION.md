# grandma.app × Skeleton — Integration Architecture

> Status: brainstorm / architecture proposal. Nothing here is committed behavior yet.
> Author context: drafted from a full read of both repos (`grandma-app` and `supa-skeleton`) on 2026-07-09.

This doc explains how to use **Skeleton** (the context substrate at `*.mother.tech`) as the long-term memory, narrative, portability, sharing, and inheritance layer underneath **grandma.app**, without changing grandma's core data model or blocking any core flow.

---

## 0. TL;DR

- grandma.app stays the **system of record for structured data** (logs, profiles, exams, vault). Skeleton becomes the **system of record for long-term semantic memory, high-level narrative documents, portability, sharing, and inheritance.**
- The plumbing already exists on both sides. grandma has `skeleton-provision-child` + `children.skeleton_space_id`; Skeleton has the **appspace key + control-grant + shadow-principal + graduation** path that was *designed* for "an external app acting as substrate for its own end-users."
- Four user-facing wins, in build order: **(1) deeper AI chat context → (2) auto-generated per-user narrative docs → (3) "your Skeleton file" export → (4) sharing & inheritance.**
- One-line mental model: **every journey (a child, a pregnancy, a fertility journey) = one Skeleton space; every log = a memory; every roll-up/insight = a versioned context doc; Guru Grandma reads both.**

---

## 1. What each side actually provides (grounded in code)

### grandma.app (the consumer app)

| Layer | Reality today |
|---|---|
| AI chat | `supabase/functions/grandma-chat/index.ts` assembles the system prompt server-side from `profiles` + the **last ~8 days** of `cycle_logs`/`pregnancy_logs`/`child_logs`, plus hardcoded topic-knowledge blocks. Model `claude-sonnet-4-5`. Returns `{reply, suggestions}`. Legacy `nana-chat` is stateless (context passed from client). |
| Insights | `generate-insights` reads the **last 30 days** of logs → Claude → writes rows into `insights` (`type ∈ pattern/trend/upcoming/nudge`). Falls back to hardcoded starter insights when there's no data. |
| Structured data | `children`, `child_logs`, `cycle_logs`, `pregnancy_logs`, `profiles`, `exams` (with AI-`extracted` jsonb), `vault_documents`, `care_circle`/`child_caregivers`, `insights`. |
| Memory horizon | **Short.** Chat sees 8 days; insights see 30. There is **no long-term semantic recall** ("last winter she had this same cough"). |
| Portability | **None.** No export/DSAR, no shareable journey artifact. |
| Existing Skeleton hook | `skeleton-provision-child` edge fn (holds `SKELETON_API_KEY`, server-only) → `POST /api/developer/spaces:provision {blueprint_id:'child-space', owner_email, label}`; stores `children.skeleton_space_id` + `skeleton_owner_principal_id`. `lib/skeletonSync.ts:provisionChildSpace()` is best-effort, never throws. **This is the beachhead.** |

### Skeleton (the substrate)

| Primitive | What it is | Why grandma wants it |
|---|---|---|
| **Space** | Context container, composite PK `(user_id, id)`, created per subject. | One space per child / pregnancy / fertility journey. |
| **Memory** | Verbatim atomic fact in `skeleton_memories`, PGVector-embedded, `decay_class`/`kind` tagged, versioned + hash-chained audit. | Every grandma log becomes a retrievable memory with **full-history** semantic recall. |
| **Context doc** | Markdown/HTML/folder/attachment in `contexts`, versioned, shareable, `subtype` (`blueprint`/`skill`), public links. | The "high-level documents generated by grandma data" the product vision calls for. |
| **Blueprint** | Template that seeds a space's folders + Agent contract + starter docs on provision. | `child-space` (exists), plus `pregnancy-journey`, `fertility-journey`. |
| **Appspace key + control-grant** | Server-to-server auth: `Bearer sk_appspace_… + ?space_id` → resolves a user-consented `control_grants` row → impersonates the space owner, scoped `read/write/full`. Accepted on `/api/memories` and `/api/contexts`. | grandma's edge functions read/write on behalf of users **without per-user OAuth**. |
| **Shadow principal** | `POST /api/developer/users:ensure {email}` mints a claimable `auth` user; the parent owns child spaces before they ever open Skeleton. **Graduation** flow (Keep/Fork/Leave) when they later sign up for real. | Powers inheritance and "export becomes a real account" with zero friction up front. |
| **Export** | `POST /api/me/export` → whole-account DSAR bundle (profile, spaces, contexts+versions, memories+versions) as a signed URL. | "Your Skeleton file." |
| **Sharing** | Per-context `share-requests` + `shares` + **public links** (`/public/c/{token}`, no account needed); space-level shares + control-grants (wildcard `*` = all spaces). | Share a pregnancy journey with a friend; wire `care_circle` to space ACLs. |
| **Client-supplied embeddings** | `save_memory`/`POST /api/memories` accept `embedding: list[float]`; search compares within one `(model, dim)` space. | grandma can embed on its own side, or let Skeleton embed — our choice. |

**Key invariant to respect:** no Supabase service-role impersonation is exposed to external apps. Every cross-user action is gated by a **user-consented control grant**. Provisioning a space via the appspace key auto-creates that grant (scope `full`) for the shadow-principal owner, so grandma's edge functions are covered — but a real, claimed user decides Keep/Fork/Leave at graduation.

## 2. Core object mapping

```
grandma.app                         Skeleton
-----------                         --------
parent (profiles.id / auth user) →  shadow principal (users:ensure by email) → claimable real account
a child (children.id)            →  space  (blueprint: child-space)      [already provisioned]
a pregnancy (behavior=pregnancy) →  space  (blueprint: pregnancy-journey)
a fertility journey (pre-preg)   →  space  (blueprint: fertility-journey)
one log row (child/cycle/preg)   →  memory (verbatim + tags + decay_class/kind)
exam / vault doc                 →  memory + context attachment (references Supabase storage)
insight / roll-up / summary      →  context doc (markdown, versioned)   ← "high-level thoughts"
care_circle member               →  space share (scope read/write) / context share (doctor → read-only)
```

**Space topology decision (recommended: per-subject spaces).** One space per child / pregnancy / fertility journey rather than one big "grandma space." Rationale: it's what the `child-space` blueprint already assumes, and it's the unit the sharing + inheritance stories operate on ("share *this pregnancy*", "hand my kid *their* space"). The parent (shadow principal) owns all of them, so cross-journey continuity is still available via `cross_space_search`.

**What becomes a memory vs a context doc.** Raw log rows → **memories** (atomic, embeddable, semantically retrievable). Human-readable roll-ups/summaries/insights → **context docs** (versioned, shareable, the artifact a human reads or inherits). This split is the whole design in one sentence.

**Source-of-truth rule.** grandma Supabase remains canonical for structured logs; Skeleton is a **derived mirror** for memories and the **canonical home** for narrative docs + portability. If they ever disagree, structured logs win and the memory is re-mirrored.

## 3. The four priorities

### Priority 1 — Deeper AI chat context (nearest-term, highest-frequency payoff)

**Problem:** `grandma-chat` only sees 8 days. It can't remember that the baby had the same rash in the spring, or that the mother said during pregnancy she was anxious about sleep.

**Design:** enrich `grandma-chat` with Skeleton retrieval before building the prompt.

```
grandma-chat edge fn (holds SKELETON_API_KEY)
  1. resolve the active space_id (child.skeleton_space_id, or the pregnancy/fertility space)
  2. GET /api/memories?search=<last user message>&space_id=<space>   (semantic, full-history)
  3. GET /api/contexts?space_id=<space>  → pull the stable "Health Summary" / "Allergies & Medical" docs
  4. inject top-k memories + the summary docs into the existing `## CONTEXT` block
  5. keep the current 8-day structured window too (recency + structure)
```

Result: Guru Grandma gains **long-horizon, semantic memory** on top of the current recency window, with almost no change to the prompt architecture — it's one more section in `## CONTEXT`. This is the single most valuable and lowest-risk integration.

**Prereq:** the active space exists and logs are mirrored as memories (Priority 2's write path). Backfill once, then mirror on write.

### Priority 2 — Per-user Skeleton space + auto-generated narrative docs

**Provisioning (extend the beachhead):**
- Children: already handled by `provisionChildSpace`. Fire it on child creation / onboarding finish (it's idempotent).
- Add `provisionPregnancySpace()` and `provisionFertilitySpace()` — same edge-function pattern, new blueprints (`pregnancy-journey`, `fertility-journey`). Store the returned `space_id` on `behaviors` (or a small `journey_spaces` table) the way `children.skeleton_space_id` works today.

**Log → memory mirror (the write path):**
- New edge function `skeleton-mirror-log` (appspace key). On each `child_logs`/`cycle_logs`/`pregnancy_logs` insert (or a batched sweep), `POST /api/memories` with `?space_id=<space>`: content = a human sentence ("2026-07-08 — feeding, 120ml, note: fussy after"), `tags=[type, date]`, `kind='event'`, `decay_class='operational'`. Use `idempotency_key = log row id` so re-runs don't duplicate.
- Embeddings: **start by letting Skeleton embed** (no embedding infra in grandma today). Revisit client-supplied embeddings only if volume/cost demands it — the API supports it either way.

**Narrative docs (the "high-level thoughts"):**
- Repurpose the `generate-insights` pattern into `skeleton-sync-docs`: read a journey's logs, ask Claude to compose/refresh a small set of **context docs** per space and `PUT /api/contexts`:
  - Child: *Health Summary*, *Feeding & Nutrition*, *Sleep*, *Milestones & Development*, *Allergies & Medical*, *Growth*.
  - Pregnancy: *Journey So Far*, *Symptoms & Mood*, *Appointments & Tests*, *Birth Plan*.
  - Fertility: *Cycle Patterns*, *TTC Timeline*.
- Because context docs are **versioned + hash-chained**, grandma gets a tamper-evident, time-stamped medical/development narrative for free — a real asset for a health app (exams, vaccines, allergies).

### Priority 3 — Export / "your Skeleton file"

**The vision:** a parent (or an adult child) can take their whole organized history — exams, allergies, foods, memories, high-level thoughts — as a portable file.

**Design:**
- The narrative docs + memories in a space *are* that file. Two delivery options:
  1. **Claim + DSAR (recommended, real portability):** parent claims the shadow principal (graduation) → runs `POST /api/me/export` → signed-URL bundle of every space's contexts+versions+memories. This makes them a first-class Skeleton owner and is GDPR Art. 15/20 compliant out of the box.
  2. **In-app bundle (no claim needed):** grandma reads a space's context docs (markdown) + memories via the appspace key and packages a human-readable folder (markdown + JSON + linked exam photos). Good for a "download my journey" button without asking anyone to sign up.
- Recommend shipping (2) first (frictionless), and offering (1) as "upgrade to a full Skeleton account you own."

### Priority 4 — Sharing & inheritance

**Share a pregnancy journey with a pregnant friend:**
- The pregnancy space already holds a *Journey So Far* context doc. "Share my journey" → `POST /api/contexts/{id}/public-link` → hand the friend a `/public/c/{token}` URL. **Read-only, no account required.** For a friend who *does* have grandma/Skeleton, use a `share-request` instead so it lands in their space.

**Inheritance — mom's 10 years handed to the grown child:**
- This is the story Skeleton's shadow-principal + graduation design was practically built for. The child's space was owned by the parent's principal the whole time; the child's entire logged history (exams, allergies, foods, memories, growth, milestones) already lives there as memories + versioned docs.
- Hand-off: when the child is old enough and signs up, `users:ensure` their email, then **transfer or share** the child-space to the child's account (space share with `full`, or ownership move). The child opens Skeleton and their whole life-from-birth is instantly explorable and searchable. No migration, no export/import dance.

**Care-circle ↔ space ACL (bonus, falls out of the same mechanism):**
- grandma's `care_circle` roles (partner/nanny/family/doctor) + `permissions[]` map cleanly onto Skeleton **space shares** and **context shares** with scope. A nanny gets read/write on the child space; a doctor gets read-only on the *Allergies & Medical* doc. This unifies grandma's bespoke permission model onto Skeleton's audited ACL.

## 4. Extra high-level ideas worth stealing

1. **One continuous life narrative across modes.** As a user moves pre-pregnancy → pregnancy → kids, Skeleton keeps the whole arc. Guru Grandma can say "during pregnancy you were anxious about sleep — here's how it's going now." The mode switch in grandma is a UI concern; the substrate never forgets the arc.
2. **A parent space, not just child spaces.** Give the *parent* their own space (their health, preferences, emotional state, recurring worries). Turns Guru Grandma from a lookup tool into a companion with continuity about the caregiver themselves — high emotional-retention value.
3. **Full-history proactive insights.** `generate-insights` is capped at 30 days. Skeleton retrieval enables genuine pattern detection across years ("every teething episode came with a fever spike").
4. **Tamper-evident medical timeline.** Skeleton's hash-chained version audit gives vaccines/exams/allergies a defensible, ordered provenance trail — a genuine trust feature for a health product, and a differentiator vs. note-taking competitors.
5. **Provider/entity canonicalization.** Skeleton's entity resolution can canonicalize pediatricians/providers/caregivers across a family's data so "Dr. Silva" is one entity everywhere.
6. **Portability as a marketing story.** "You own your data and can take it with you" is rare in parenting apps and is literally a built-in Skeleton endpoint.

> Deliberately **out of scope / flagged:** any "moms like you" cross-user sharing of health data. Skeleton keeps memories per-space and PHI-sensitive; cross-user knowledge would need a separate, privacy-reviewed design. Don't let it ride in on this integration.

## 5. Phased build plan

| Phase | Ships | New/changed pieces | User-visible win |
|---|---|---|---|
| **0 (done)** | Child space provisioning scaffold | `skeleton-provision-child`, `children.skeleton_space_id` | — |
| **1** | Chat context enrichment | Provision active-journey spaces; `skeleton-mirror-log` (backfill + on-write); `grandma-chat` reads `/api/memories` + `/api/contexts` | Guru Grandma remembers everything, not just 8 days |
| **2** | Narrative docs | `skeleton-sync-docs` (Claude → versioned context docs per space); `pregnancy-journey` + `fertility-journey` blueprints | Auto-written "high-level" health/journey docs |
| **3** | Export | In-app journey bundle (appspace read) + optional claim → `/api/me/export` | "Download / own your data" |
| **4** | Sharing & inheritance | Context public-links; `care_circle` → space/context shares; child-space hand-off at graduation | Share a journey; pass a child's history down |

**Suggested first slice to prototype:** Phase 1 for a single child. Provision (already works) → mirror that child's `child_logs` as memories → add one retrieval call to `grandma-chat`. It's small, isolated, and demonstrates the whole value proposition end to end.

## 6. Decisions needed (call these before building)

1. **Consent & PHI.** Storing child/pregnancy health data in a third-party substrate needs an explicit, logged opt-in in grandma onboarding, mapped to Skeleton's control-grant consent. Legal review before Phase 1 touches real users.
2. **Space granularity.** Confirm per-subject spaces (child / pregnancy / fertility) vs one space per user. This doc recommends per-subject.
3. **Write path.** Ongoing writes via `/api/memories` + `/api/contexts` with the appspace bearer + `?space_id` (covered by the provision-time control grant), vs the narrower `/api/developer/spaces/{id}/context`. Recommend the former for full memory+doc control.
4. **Embeddings.** Skeleton-side (simple, start here) vs client-supplied from grandma (control/cost at scale).
5. **Mirror cadence.** On-write (realtime, more edge-function calls) vs batched sweep (cheaper, laggy). Recommend batched nightly + on-write for high-signal events.
6. **Source of truth on conflict.** Confirm structured logs win and memories are re-derived (this doc's assumption).
7. **Shadow-principal email = parent email.** Confirm this is acceptable and matches the graduation/claim UX we want.
